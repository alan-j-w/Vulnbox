Introduction To Brute-Force Attacks

This modules focuses on brute-force and authentication attacks on web applications. Participants will use tools like Burp Suite to explore vulnerabilities in login mechanisms. The lab offers hands-on experience in testing password strength and analyzing authentication processes, teaching participants how to improve web application security against unauthorized access.

Module 1: What is a Brute-Force Attack?

A **brute-force attack** is the digital equivalent of trying every single key on a keychain until one fits the lock. In cybersecurity, it's a trial-and-error method used to guess login credentials, encryption keys, or hidden web pages.

The attacker systematically tries all possible combinations of letters, numbers, and symbols until the correct one is found. While it sounds simple, modern computers can perform thousands or even millions of guesses per second, making this a very real threat against weak passwords.

Analogy: The Combination Lock

Imagine a suitcase with a 3-digit combination lock. You've forgotten the code. A brute-force approach would be to start with 000, then try 001, 002, and so on, all the way to 999. It's tedious, but you are **guaranteed** to eventually find the correct combination.

Module 2: Types of Brute-Force Attacks

While the concept is simple, attackers use different strategies to speed up the process.

  • Simple Brute-Force Attack: This is the purest form, where the algorithm tries every single possible combination. This can be very slow for long passwords (e.g., `aaaa`, `aaab`, `aaac`...).
  • Dictionary Attack: This is a much more efficient method. Instead of trying random combinations, the attacker uses a "dictionary"â€”a large list of common words, phrases, and previously leaked passwords. An attacker's dictionary might contain millions of the most commonly used passwords like `password123`, `123456`, `qwerty`, etc.
  • Hybrid Brute-Force Attack: This combines a dictionary attack with simple brute-force logic. For example, it might take a word from a dictionary, like `password`, and then add numbers or symbols to it (`password1`, `password!`, `password2025`).

Module 3: Prevention Techniques

Brute-force attacks only work when a system allows an attacker to make unlimited, rapid guesses. Strong defenses focus on making each guess difficult or limiting the number of attempts.

Key Defenses

  • Strong Password Policies: Enforcing long passwords with a mix of uppercase letters, lowercase letters, numbers, and symbols makes a simple brute-force attack computationally impossible.
  • Rate Limiting: This is one of the most effective defenses. The system limits the number of login attempts that can be made from a single IP address in a given period (e.g., 5 failed attempts in 1 minute).
  • Account Lockout: After a certain number of failed attempts (e.g., 10), the account is temporarily locked, preventing any further guesses for a set time.
  • CAPTCHA: Requiring a user to solve a simple puzzle (like identifying images) after a few failed attempts proves they are a human, not an automated script.

Case Study: The GitHub Brute-Force Attack (2013)

The Attack

In 2013, GitHub, one of the world's largest code hosting platforms, was the target of a large-scale brute-force attack. Attackers used massive lists of usernames and passwords that had been leaked from other websites, a technique known as **credential stuffing**.

The Result

The attackers were successful in compromising a number of user accounts that had weak or reused passwords. This incident highlighted the real-world danger of brute-force attacks, even against major tech companies. In response, GitHub implemented much stronger security measures, including rate limiting and mandatory two-factor authentication, which are now standard across the industry.

Your Goal

Now that you understand the theory, it's time to perform a brute-force attack yourself. You will be presented with a login portal that has no rate limiting. Your mission is to discover the correct credentials and gain access.

How to Perform the Attack: A Step-by-Step Guide

We will use **Burp Suite**, an industry-standard tool for web application security testing, to automate the attack.

  • Step 1: Intercept the Request. With Burp Suite running, turn on the proxy's "Intercept" feature. Go to the brute-force lab page in your browser and attempt to log in with any random username and password (e.g., `admin`/`test`). Burp will capture this login request before it reaches the server.
  • Step 2: Send to Intruder. In Burp's Proxy tab, you will see the captured HTTP request. Right-click anywhere in the request and select "Send to Intruder."
  • Step 3: Set Payload Positions. Go to the Intruder tab. In the "Positions" sub-tab, Burp will have automatically guessed which parts of the request to attack. We only want to attack the password. Click the "Clear Â§" button on the right to remove all default positions. Then, highlight the value of the password parameter (e.g., the word `test` that you typed) and click the "Add Â§" button. This tells Intruder that the password is the only part of the request we want to change.
  • Step 4: Configure the Payload. Go to the "Payloads" sub-tab. Under "Payload type," select "Simple list." In the "Payload Options" box below, you can either paste a short list of common passwords or use the "Load" button to import a larger password list (like a file from SecLists or another wordlist).
  • Step 5: Launch the Attack. Click the "Start attack" button in the top right. A new window will open, and Intruder will begin sending each password from your list to the server, one by one.
  • Step 6: Analyze the Results. Watch the results table. Most attempts will have the same status code (e.g., 200 OK) and the same response length. The **correct password** will likely result in a different response, such as a different status code (like a 302 redirect) or a different response length. This anomaly indicates a successful login!
